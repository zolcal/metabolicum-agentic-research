# Memory and vector architecture

The most important architectural decision in this section is project and database isolation. The rebuilt research pipeline runs in a new standalone project, `metabolicum-agentic-research`, with its own Supabase project. The `metasync` production Supabase project is untouched by the pipeline; only finished, reviewed export artifacts cross the boundary through controlled import. This is a separate project, not just a separate schema in the same project, for four reasons. First, resource contention: agentic workflows produce write spikes — one source can mean dozens of claim inserts in seconds — and a shared project would mean user-facing queries can stall behind research writes. Second, HIPAA cleanliness: `metasync` carries user lab data, while `metabolicum-agentic-research` carries only public research content; separate projects mean separate API keys, separate row-level security policies, and zero credential overlap. Third, blast radius: a bad migration on the agentic research project corrupts research data, while a bad migration on production takes down the web app. Fourth, lifecycle independence: research database growth, schema changes, and retention policy must not be coupled to production-database operations. `[JUDGMENT]` Use the same Supabase organization for administrative convenience if desired, but keep the projects, credentials, and local filesystems distinct.

The Pinecone question, asked honestly, is what Pinecone does that pgvector cannot, at our scale. Pinecone is a managed vector database with a proprietary HNSW-derived index (Encore, retrieved 2026-05-13). The "memory" framing in some Pinecone marketing is just that — marketing. The underlying capability is fast similarity search with metadata filtering. Pinecone Inference, their hosted embedding service, and Pinecone Assistant API, their RAG-as-a-service wrapper, are separate products bundled with Pinecone Serverless. `[JUDGMENT]` Neither is a meaningful differentiator for us: Pinecone Inference duplicates what we would get from OpenAI or Voyage embeddings directly, and Pinecone Assistant API is a constrained "ask-questions-of-your-docs" surface that does not fit a multi-stage agentic research pipeline. They are useful for teams that do not want to build their own RAG stack; we already have the BiomarkerClaim schema and an opinionated council, and those abstractions do not survive being delegated to Assistant API.

What Pinecone-the-database offers compared to pgvector is zero infrastructure work, single-digit-millisecond latency at 100 million-plus vectors, and multi-tenant managed scaling. What it does not offer is better recall at equivalent settings, or anything our agents could not do with a SELECT on Supabase.

pgvector in 2026 is production-grade. Per Groovy Web's vector database comparison (retrieved 2026-05-13), "At 1M vectors, all four hit 95%+ recall with default settings. At 100M vectors, Pinecone and Weaviate maintain recall without tuning, while pgvector requires careful HNSW parameter optimization … pgvector is absolutely production-grade in 2026." With pgvectorscale, the StreamingDiskANN extension from Timescale, the numbers shift further in pgvector's favor. Per the official pgvectorscale GitHub README (github.com/timescale/pgvectorscale, retrieved 2026-05-13), PostgreSQL with pgvector and pgvectorscale can deliver strong p95 latency and query-throughput results on large benchmark datasets. The practical capacity of a single well-provisioned Postgres instance is far above the expected pilot workload.

Among alternatives, Qdrant (Rust, open source) is capable but adds a dependency we do not need. Weaviate offers strong hybrid (semantic plus keyword) search without a BM25 bolt-on but we do not need hybrid yet. Chroma is fine for local development and prototypes but is not a production vector database for multi-user workloads, per Jahanzaib (retrieved 2026-05-13). LanceDB is embeddable and interesting for offline batch use, but we skip it for now.

The orchestrator-memory question is what the user actually needs: dynamic context loading for long agentic workflows, not vector search per se. There are three patterns. The first is summarization checkpoints — at each stage boundary, we write a structured summary to state. The second is vector-recall on demand — RAG over our own pipeline outputs via pgvector. The third is an external memory service like Mem0 or Letta. `[JUDGMENT]` Patterns one and two, both running on the `metabolicum-agentic-research` Supabase project plus pgvector, are enough for the first implementation. Pattern three is an unnecessary new dependency until the implementation runner proves it needs external memory.

The recommendation is to use the `metabolicum-agentic-research` Supabase project plus pgvector for research persistence and vector recall. Two pgvector tables suffice: `signal_embeddings` holds embeddings of social-signal items, and `claim_embeddings` holds embeddings of claim text for deduplication and cross-marker discovery. The embedding model remains runtime-configurable. Prior semantic-search research should be retained as a calibration input: `intfloat/multilingual-e5-large` and an approximate 0.76 cosine threshold are known candidate baselines for multilingual practitioner language, while `text-embedding-3-large` and `voyage-3` remain acceptable cloud alternatives. The database records the model name, vector dimensions, threshold, and scoring-config version so future calibration can explain why a source was promoted or missed.

A separate but related use of the same embedding model is **semantic practitioner matching** (section 20). The practitioner index builder embeds practitioner metadata and marker topic descriptors, then uses cosine similarity to find practitioners who discuss a marker conceptually even when they do not use any known alias. This is a fallback layer, not a replacement for the thesaurus, and it runs offline as part of `scripts/collect_practitioners.py` rather than inside the database.

Embedding refresh is versioned, not destructive. When the model or threshold changes, new embeddings can coexist with old embeddings under a new `embedding_model` and `embedding_config_version`. Queries must select the active config version. Full re-embedding is an implementation decision, but old vectors are not silently overwritten because past ranking decisions must remain explainable.
